# 客户首次使用信息清单

这份文档是 `V5.1 Hardening` 的信息采集总表，解决两个问题：

1. 客户第一次使用这个 skill 时，必须先收集哪些真实信息
2. 如果后面要新增群、新增机器人、给群加 worker、删 worker、下线群，还要额外补哪些信息

## 统一入口

当前主线统一入口只有一套：

- 推荐输入模板：`references/input-template-v51-fixed-role-multi-group.json`
- canonical schema：`roleCatalog + teams(profileId + override)`
- 旧 inline `teams` 只保留兼容，不再作为主线交付规范

核心原则：

- 优先把真实值填写到结构化输入文件，而不是散落在 Codex 对话里
- `accountId` 不是飞书平台自动给你的英文名字，`accountId 是我们自己定义的键名`
- 一个输入文件可以一次配置 `1` 个群或多个群
- 多个群可以有不同的角色组合，但同一个 bot 在所有群里应保持同一个角色
- `visibleLabel` 是显示层真源，群里标题和 runtime manifest 都从它派生

## 第一次交付必须先收集的 7 类信息

### 1. 部署环境信息

- 服务器地址或 SSH 别名
- 登录用户
- `~/.openclaw/openclaw.json` 实际路径
- OpenClaw 版本
- 平台类型：`linux / macos / wsl2`
- service manager：`systemd --user / launchd / manual`
- 是否已经存在老配置，是否必须 brownfield incremental

### 2. 每个飞书机器人应用信息

每个 bot 至少收集：

- `appId`
- `appSecret`
- `encryptKey`
- `verificationToken`
- 我们自己定义的 `accountId`

推荐命名：

- `supervisor_bot`
- `ops_bot`
- `finance_bot`

如果客户已经有自己的命名，也可以直接沿用，但必须全局唯一。

### 3. 每个群的真实信息

每个群至少收集：

- 群名称
- 群 `peerId/chat_id`
- 是否必须 `@`
- 这个群要启用哪些角色
- 这个群的 `displayName`
- 这个群是否是 canary 群 / 正式群

### 4. 全局固定 bot-role 映射

先定清楚：

- 哪个 bot 永远做 `supervisor`
- 哪个 bot 永远做 `ops`
- 哪个 bot 永远做 `finance`
- 如果扩展更多角色，哪个 bot 永远做该角色

不要让同一个 bot 在不同群切换不同角色。

### 5. 角色目录信息

对每个 `roleCatalog.<profileId>` 至少要明确：

- `kind`
- `accountId`
- `name`
- `role`
- `visibleLabel`
- `description`
- `responsibility`
- `identity`
- `mentionPatterns`
- `visibility`
- `systemPrompt`

### 6. 每个群的团队配置

每个 `team` 至少要明确：

- `teamKey`
- `displayName`
- `group.peerId`
- `group.entryAccountId`
- `group.requireMention`
- `supervisor.profileId`
- `supervisor.agentId`
- `workers[]`
- `workflow.stages`

### 7. 验收与回滚信息

- canary 测试群
- 内部测试提示词
- 外部测试提示词
- 回滚负责人
- 备份路径
- 谁来执行 `openclaw config validate / gateway restart / canary`

## 这些信息填写到哪里

推荐全部填写到：

- `references/input-template-v51-fixed-role-multi-group.json`

字段映射如下。

### 账号信息

- `accounts[].accountId`
- `accounts[].appId`
- `accounts[].appSecret`
- `accounts[].encryptKey`
- `accounts[].verificationToken`

### 群与团队信息

- `teams[].teamKey`
- `teams[].displayName`
- `teams[].group.peerId`
- `teams[].group.entryAccountId`
- `teams[].group.requireMention`

### 角色目录信息

- `roleCatalog.<profileId>.kind`
- `roleCatalog.<profileId>.accountId`
- `roleCatalog.<profileId>.name`
- `roleCatalog.<profileId>.role`
- `roleCatalog.<profileId>.visibleLabel`
- `roleCatalog.<profileId>.description`
- `roleCatalog.<profileId>.responsibility`
- `roleCatalog.<profileId>.identity`
- `roleCatalog.<profileId>.mentionPatterns`
- `roleCatalog.<profileId>.visibility`
- `roleCatalog.<profileId>.systemPrompt`

### 主管信息

- `teams[].supervisor.agentId`
- `teams[].supervisor.profileId`
- `teams[].supervisor.visibleLabel`（如需覆盖）
- `teams[].supervisor.systemPrompt`（如需覆盖）

### worker 信息

- `teams[].workers[].agentId`
- `teams[].workers[].profileId`
- `teams[].workers[].visibleLabel`（如需覆盖）
- `teams[].workers[].accountId`（如需覆盖）
- `teams[].workers[].visibility`（如需覆盖）
- `teams[].workers[].description`（如需覆盖）
- `teams[].workers[].systemPrompt`（如需覆盖）

### 调度顺序

- `teams[].workflow.mode`
- `teams[].workflow.stages`

## 第一次交付怎么填

### 只配置 1 个群

- `teams[]` 里只保留一个对象
- 该群启用什么角色，就只在该 `team.workers[]` 下写什么角色
- `workflow.stages` 只声明这个群当前启用的 worker

### 一次配置多个群

- `teams[]` 里写多个对象
- 每个对象对应一个群
- 每个群都可以有自己的角色组合、描述和提示词

## 扩容与裁剪时额外要收集什么

### 新增一个群

额外收集：

- 新群 `peerId`
- 新群 `displayName`
- 新群 `teamKey`
- 新群启用哪些角色
- 新群 supervisor 是复用现有 profile 还是新建 profile
- 新群是否需要独立的 supervisor prompt / identity / visibleLabel

要修改的位置：

- `teams[]` 新增一个 team
- 如需新 supervisor prompt，则在 `roleCatalog` 新增新的 supervisor profile
- `workflow.stages` 按该群实际 worker 顺序写全

### 新增一个机器人账号

额外收集：

- 新 bot 的 `accountId / appId / appSecret / encryptKey / verificationToken`
- 该 bot 固定承担哪个角色
- 它要在哪些群里启用

要修改的位置：

- `accounts[]`
- `roleCatalog.<profileId>.accountId`
- 对应 `teams[].workers[]` 或 `teams[].supervisor`

### 给现有群增加一个 worker

额外收集：

- 新 worker 用哪个 `profileId`
- 是否复用现有 bot
- 群里显示名是否要覆盖
- 新 worker 在 `workflow.stages` 里的顺序

要修改的位置：

- `teams[].workers[]`
- `teams[].workflow.stages`
- 若是新角色，还要在 `roleCatalog` 新建 profile

### 从现有群移除一个 worker

额外收集：

- 要删除哪个 `agentId`
- 这个 worker 是否还会在其他群继续使用
- 删除后新的 `workflow.stages` 顺序

要修改的位置：

- 从该 `team.workers[]` 删除
- 从该 `team.workflow.stages` 删除
- 如果这个 profile 不再被任何 team 引用，可以考虑同步从 `roleCatalog` 删除

### 下线一个群

额外收集：

- 要下线的 `teamKey`
- 是否立即停用 watchdog
- 该群 runtime 数据是否要保留归档

要修改的位置：

- 从 `teams[]` 删除对应 team
- 清理对应 runtime manifest / watchdog / SQLite / workspace

## 推荐交付包

第一次交付时，不要只给客户一段聊天提示词。推荐同时交付：

1. 输入文件  
`customer-v51-prod-input.json`

2. 产品手册
`codex-prompt-templates-v51-team-orchestrator.md`

3. Codex 执行提示词
`客户首次使用-Codex提示词.md`

4. 真实案例
`客户首次使用真实案例.md`

## 最后检查

在把输入文件交给 Codex 前，至少确认：

- `appId / appSecret / encryptKey / verificationToken` 已补齐
- 每个群的 `peerId` 已确认真实可用
- 每个角色引用的 `accountId` 与 `accounts[]` 里的键名一致
- `workflow.stages` 与 `workers[]` 完全一致
- 一个群里没有让同一个 bot 同时承担两个可见角色
- 新增 worker 时，`workflow.stages` 没漏写也没重复写
- 删除 worker 时，`workflow.stages` 已同步删掉
