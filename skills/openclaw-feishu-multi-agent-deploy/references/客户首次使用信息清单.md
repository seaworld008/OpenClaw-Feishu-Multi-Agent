# 客户首次使用信息清单

这份文档解决两个问题：

1. 客户第一次使用这个 skill 时，必须先收集哪些真实信息  
2. 这些信息应该填写到哪里，而不是临时散落在聊天里

## 核心原则

- 优先把真实值填写到结构化输入文件，而不是直接散落在 Codex 对话中
- 推荐输入模板：`references/input-template-v5-fixed-role-multi-group.json`
- `accountId` 不是飞书平台自动给你的“英文名字”，`accountId 是我们自己定义的键名`
- 一个输入文件可以一次配置 `1` 个群或多个群
- 多个群可以有不同的角色组合，但同一个 bot 在所有群里应保持同一个角色

## 必须先收集的 6 类信息

### 1. 部署环境信息

- 服务器地址或 SSH 别名
- 登录用户
- `~/.openclaw/openclaw.json` 实际路径
- OpenClaw 版本
- 平台类型：`linux / macos / wsl2`
- service manager：`systemd --user / launchd / manual`

### 2. 每个飞书机器人应用信息

每个 bot 至少收集：

- `appId`
- `appSecret`
- `encryptKey`
- `verificationToken`
- 我们自己定义的 `accountId`

推荐命名：

- `supervisor_bot`
- `ops_bot`
- `finance_bot`

如果客户已经有自己的命名，也可以直接沿用，但必须全局唯一。

### 3. 每个群的真实信息

每个群至少收集：

- 群名称
- 群 `peerId/chat_id`
- 是否必须 `@`
- 这个群要启用哪些角色

### 4. 全局固定角色映射

先定清楚：

- 哪个 bot 永远做 `supervisor`
- 哪个 bot 永远做 `ops`
- 哪个 bot 永远做 `finance`

如果后续扩展更多角色，也先定成全局映射，不要让同一个 bot 在不同群切换不同角色。

### 5. 每个群的团队配置

每个群至少要明确：

- `teamKey`
- `displayName`
- `group.peerId`
- `group.entryAccountId`
- `supervisor.systemPrompt`
- `workers[]`
- `workflow.stages`

### 6. 验收与回滚信息

- canary 测试群
- 内部测试提示词
- 外部测试提示词
- 回滚负责人
- 备份路径

## 这些信息填写到哪里

推荐全部填写到：

- `references/input-template-v5-fixed-role-multi-group.json`

字段映射如下：

### 账号信息

- `accounts[].accountId`
- `accounts[].appId`
- `accounts[].appSecret`
- `accounts[].encryptKey`
- `accounts[].verificationToken`

### 群与团队信息

- `teams[].teamKey`
- `teams[].displayName`
- `teams[].group.peerId`
- `teams[].group.entryAccountId`
- `teams[].group.requireMention`

### 主管信息

- `teams[].supervisor.agentId`
- `teams[].supervisor.name`
- `teams[].supervisor.description`
- `teams[].supervisor.role`
- `teams[].supervisor.responsibility`
- `teams[].supervisor.systemPrompt`

### worker 信息

- `teams[].workers[].agentId`
- `teams[].workers[].accountId`
- `teams[].workers[].name`
- `teams[].workers[].description`
- `teams[].workers[].role`
- `teams[].workers[].responsibility`
- `teams[].workers[].visibility`
- `teams[].workers[].systemPrompt`

### 调度顺序

- `teams[].workflow.mode`
- `teams[].workflow.stages`

## 一个群和多个群怎么填

### 只配置 1 个群

- `teams[]` 里只保留一个对象
- 这个群启用什么角色，就只在该 `team.workers[]` 下写什么角色

### 一次配置多个群

- `teams[]` 里写多个对象
- 每个对象对应一个群
- 每个群都可以有自己的角色组合、描述和提示词

## 推荐交付方式

第一次交付时，不要只给客户一段聊天提示词。推荐同时交付：

1. 输入文件  
`customer-prod-input.json`

2. Codex 执行提示词  
`客户首次使用-Codex提示词.md`

3. 真实案例  
`客户首次使用真实案例.md`

## 最后检查

在把输入文件交给 Codex 前，至少确认：

- `appId / appSecret / encryptKey / verificationToken` 已补齐
- 每个群的 `peerId` 已确认真实可用
- 每个角色引用的 `accountId` 与 `accounts[]` 里的键名一致
- `workflow.stages` 与 `workers[]` 完全一致
- 一个群里没有让同一个 bot 同时承担两个可见角色
