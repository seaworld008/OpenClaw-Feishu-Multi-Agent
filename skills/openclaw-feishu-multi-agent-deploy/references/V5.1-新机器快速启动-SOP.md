# V5.1 新机器快速启动 SOP

适用目标：

- 一台全新的 `Linux / macOS / Windows + WSL2` 电脑
- 需要把 `V5.1 Hardening` 从 0 拉起来
- 需要让客户第一次就按固定模板收集信息、生成配置、上线验证
- 目标效果：一个或多个飞书群，每个群固定 `1` 个 supervisor + `N` 个 worker，群与群隔离；worker 可串行或并行分析，但群里消息统一顺序发布，主管最后统一收口

不适用目标：

- Windows 原生 service 路线
- 任意 mesh 式自由协作
- 同一个 bot 在同一个群里承担多个可见角色

统一入口：

- 产品手册：`references/codex-prompt-templates-v51-team-orchestrator.md`
- 真实案例：`references/客户首次使用真实案例.md`
- 收集清单：`references/客户首次使用信息清单.md`
- 操作型提示词：`references/客户首次使用-Codex提示词.md`

当前主线路径统一按下面 4 条理解：

- 正常路径：ingress -> controller -> outbox -> sender -> callback sink
- ingress transcript 扫描仅用于建单 repair；callback 不再走 hidden main / transcript recovery
- `teamKey` 是唯一内部隔离主键；`group peerId` 只是入口地址
- 插件与 OpenClaw 之间只依赖窄 adapter
- 若一个 stage 内有多个 worker，则它们可以并行分析，但 `progress/final` 必须按 `publishOrder` 顺序发群

---

## 0. 你先要搞清楚“哪台机器做什么”

推荐把机器分成两类：

1. 操作机（Operator Machine）
- 你本人使用 Codex 的电脑
- 用来安装这个 skill、准备输入文件、发 Codex 提示词、远程部署

2. 运行机（Gateway Machine）
- 真正运行 OpenClaw Gateway 的机器
- 真正连接飞书的机器
- 真正保存 `~/.openclaw/openclaw.json`、session、SQLite、manifest、watchdog 的机器

推荐生产实践：

- 你的笔记本 = 操作机
- 一台 Linux / WSL2 / macOS 机器 = 运行机

如果是小规模测试，也可以同一台机器同时承担两个角色。

本文默认：

- 你当前在操作机上
- 你要通过 SSH 部署到运行机

## 1. 部署前必须收集的真实信息

如果这些信息不齐，不要开始部署。

### 1.1 运行机信息

至少准备：

- 运行机地址或 SSH 别名
- SSH 用户名
- 是否是 `linux / macos / wsl2`
- 运行机上 `~/.openclaw` 是否允许写入
- 运行机是否已经安装过 OpenClaw

推荐先在操作机上配置一个 SSH 别名：

```sshconfig
Host openclaw-prod
  HostName 1.2.3.4
  User seaworld
  Port 22
```

这样后面所有命令都可以直接用：

```bash
ssh openclaw-prod
scp <本地文件> openclaw-prod:<远端路径>
```

### 1.2 每个飞书 bot 的应用信息

每个 bot 至少收集：

- `appId`
- `appSecret`
- `encryptKey`
- `verificationToken`
- 我们自己定义的 `accountId`

强约束：

- `accountId` 不是飞书自动给你的英文名
- `accountId` 是我们在 OpenClaw 配置里自己定义的稳定键名
- 推荐直接按角色命名：
  - `supervisor_bot`
  - `ops_bot`
  - `finance_bot`

### 1.3 每个目标群信息

每个群至少收集：

- 群名称
- 群真实 `peerId/chat_id`
- 是否要求必须 `@`
- 这个群启用哪些角色

如果你不知道群 `peerId`，先用下面的方法取：

1. 把 bot 拉进群
2. 群里 `@机器人` 发一条测试消息
3. 在运行机上看日志：

```bash
openclaw logs --follow
```

4. 从入站事件里找 `peer.id`
5. 群聊一般就是 `oc_xxx`

### 1.4 全局固定 bot-role 映射

先定死，不要边部署边改：

- 哪个 bot 永远是 `supervisor`
- 哪个 bot 永远是 `ops`
- 哪个 bot 永远是 `finance`

生产推荐：

- 同一个 bot 可以跨很多群复用
- 但它在所有群里都保持同一个角色

### 1.5 每个群的提示词和职责

每个群都要单独写：

- supervisor 的 `systemPrompt`
- 每个 worker 的 `systemPrompt`
- `description`
- `role`
- `responsibility`

不要偷懒把所有群写成同一段 prompt。可以复用骨架，但要保留群级差异。

## 2. 在运行机上安装 OpenClaw

以下命令基于 OpenClaw 官方文档当前写法整理，按 `2026-03-09` 验证。

### 2.1 基础要求

OpenClaw 官方当前要求：

- Node `22+`

先检查：

```bash
node --version
npm --version
```

如果 Node 不满足要求，先安装或升级 Node，再继续。

### 2.2 安装 OpenClaw

官方推荐快速安装：

```bash
curl -fsSL https://openclaw.ai/install.sh | bash
```

如果你习惯用 npm 全局安装，也可以：

```bash
npm install -g openclaw@latest
```

安装后先确认命令可用：

```bash
openclaw --version
openclaw doctor
```

如果提示 `openclaw: command not found`，优先修正 PATH：

```bash
export PATH="$(npm prefix -g)/bin:$PATH"
```

然后重新开一个 shell，再执行：

```bash
openclaw --version
```

### 2.3 安装 Gateway 服务

官方推荐：

```bash
openclaw onboard --install-daemon
```

这个命令会帮你完成：

- Gateway 服务安装
- 本地运行目录初始化
- 基础配置引导

安装完成后检查：

```bash
openclaw gateway status
openclaw status
openclaw dashboard
```

如果是 Linux / WSL2，也可以再确认 systemd user service：

```bash
systemctl --user status openclaw-gateway.service
```

如果是 macOS，确认 LaunchAgent 即可。

### 2.4 安装飞书插件

官方插件安装命令：

```bash
openclaw plugins install @openclaw/feishu
```

然后检查：

```bash
openclaw plugins list
openclaw plugins doctor
```

预期：

- 能看到 `@openclaw/feishu`
- 没有明显的 plugin load error

## 4. 准备客户输入文件

这是整个流程里最关键的一步。真实值优先写入输入文件，不要散落在聊天里。

### 4.1 复制正式模板

在操作机仓库根目录执行：

```bash
mkdir -p ~/deployments
cp skills/openclaw-feishu-multi-agent-deploy/references/input-template-v51-fixed-role-multi-group.json \
  ~/deployments/customer-v51-prod-input.json
```

### 4.2 填写输入文件

至少需要填这几个层级：

- `accounts[]`
- `roleCatalog`
- `teams[]`
- `teams[].group`
- `teams[].supervisor`
- `teams[].workers[]`
- `teams[].workflow.stages`

当前主线推荐 schema：

- 根层用 `roleCatalog` 统一维护 `name / role / visibleLabel / description / responsibility / systemPrompt`
- `teams[].supervisor` 与 `teams[].workers[]` 只写 `profileId + agentId + override`
- worker / supervisor 的群内标题统一从 `visibleLabel` 派生，例如 `【{visibleLabel}进度｜TG-xxxx】`
- `workflow.stages` 现在支持 `serial` 和 `parallel` stage group
- 如果一个 stage 内有多个 worker 并行分析，必须显式写 `stageKey + agents + publishOrder`

最小结构示例：

```json
{
  "mode": "plugin",
  "defaultAccount": "supervisor_bot",
  "connectionMode": "websocket",
  "domain": "feishu",
  "accounts": [
    {
      "accountId": "supervisor_bot",
      "appId": "cli_xxx",
      "appSecret": "xxx",
      "encryptKey": "xxx",
      "verificationToken": "xxx"
    },
    {
      "accountId": "ops_bot",
      "appId": "cli_yyy",
      "appSecret": "yyy",
      "encryptKey": "yyy",
      "verificationToken": "yyy"
    },
    {
      "accountId": "finance_bot",
      "appId": "cli_zzz",
      "appSecret": "zzz",
      "encryptKey": "zzz",
      "verificationToken": "zzz"
    }
  ],
  "roleCatalog": {
    "supervisor_default": {
      "kind": "supervisor",
      "roleKey": "supervisor",
      "accountId": "supervisor_bot",
      "name": "主管机器人",
      "role": "主管总控",
      "visibleLabel": "主管",

      "responsibility": "接单、拆解、调度、统一收口",

    },
    "ops_default": {
      "kind": "worker",
      "roleKey": "ops",
      "accountId": "ops_bot",
      "name": "运营机器人",
      "role": "运营专家",
      "visibleLabel": "运营",
      "description": "负责活动、节奏和执行动作。",
      "responsibility": "活动打法、节奏、执行动作",
      "visibility": "visible",
      "systemPrompt": "填写当前群的运营提示词"
    },
    "finance_default": {
      "kind": "worker",
      "roleKey": "finance",
      "accountId": "finance_bot",
      "name": "财务机器人",
      "role": "财务专家",
      "visibleLabel": "财务",
      "description": "负责预算、毛利、ROI 与风险。",
      "responsibility": "预算、毛利、ROI 与风险",
      "visibility": "visible",
      "systemPrompt": "填写当前群的财务提示词"
    }
  },
  "teams": [
    {
      "teamKey": "internal_main",
      "displayName": "内部生产团队群",
      "group": {
        "peerId": "oc_xxx",
        "entryAccountId": "supervisor_bot",
        "requireMention": true
      },
      "supervisor": {
        "profileId": "supervisor_default",
        "agentId": "supervisor_internal_main"
      },
      "workers": [
        {
          "profileId": "ops_default",
          "agentId": "ops_internal_main"
        },

              "ops_internal_main",
```

这段的真实语义是：

- `ops_internal_main` 和 `finance_internal_main` 可以同时分析
- 但群里一定先发运营，再发财务
- 两个 worker 都发布完成后，主管才进入最终统一收口

如果你需要看带真实双群、真实机器人和真实提示词的完整版本，不要在这份 SOP 里继续拼接，直接去看：

- `references/客户首次使用真实案例.md`
- `references/codex-prompt-templates-v51-team-orchestrator.md`

### 4.3 填写时最容易犯的错

部署前逐条确认：

- 不要把 `appId` 写到 `accountId`
- 不要让同一个 bot 在同一个群里承担两个可见角色
- 不要让 `workflow.stages` 漏掉 worker
- 不要让 `workflow.stages` 和 `workers[]` 不一致
- 不要把不同群写成同一个 `peerId`
- 不要把所有群的 `systemPrompt` 写成完全一样

## 5. 先本地生成一次配置产物

在操作机仓库根目录执行：

```bash
python3 skills/openclaw-feishu-multi-agent-deploy/scripts/v51_team_orchestrator_deploy.py \
  --input ~/deployments/customer-v51-prod-input.json \
  --out ~/deployments/generated-v51 \
  --openclaw-home ~/.openclaw \
  --systemd-user-dir ~/.config/systemd/user \
  --mode plugin
```

你应该能拿到一组生成产物，用于预览：

- patch/snippets
- summary
- `openclaw-feishu-plugin-v51-runtime-latest.json`
- `~/.openclaw/v51-runtime-manifest.json`
- `~/.config/systemd/user/v51-team-*.service/.timer`
- `~/.openclaw/teams/<teamKey>/workspaces/...` 下的 role-specific 契约文件（`AGENTS.md / SOUL.md / USER.md / IDENTITY.md / TOOLS.md / HEARTBEAT.md`）
- 后续真实运行时，控制面会在成功消费 worker callback 后自动轮转 supervisor hidden main 和该 worker 的 main session；这是预期的稳定化动作

这一步不是替代 Codex，而是先发现输入文件是否明显有问题。
如果这一步跑完后 team workspace 里还保留默认 `BOOTSTRAP.md`，说明部署不完整，不能直接开始群测。

## 7. 运行机上必须执行的上线后检查

Codex 部署结束后，在运行机执行：

```bash
openclaw config validate
openclaw gateway restart
openclaw agents list --bindings
openclaw channels status --probe
openclaw health
```

如果是 Linux / WSL2，再执行：

```bash
systemctl --user status openclaw-gateway.service --no-pager
systemctl --user list-timers --all --no-pager | grep openclaw || true
```

你应该看到：

- 配置合法
- Gateway 正常运行
- 飞书账号探针可用

如果这是测试环境，且之前已经跑过旧版 team workspace，推荐在第一轮群测前做一次 clean redeploy：

1. 备份 `~/.openclaw/openclaw.json`、`~/.openclaw/v51-runtime-manifest.json`、`~/.openclaw/teams/<teamKey>/workspaces`
2. 重新执行一次 `v51_team_orchestrator_deploy.py --openclaw-home ~/.openclaw`
3. 确认 `~/.openclaw/teams/<teamKey>/workspaces/...` 下已经是 role-specific `AGENTS.md / SOUL.md / USER.md / IDENTITY.md / TOOLS.md / HEARTBEAT.md`
4. 确认默认 `BOOTSTRAP.md` 已不存在
5. 再执行 `v51_team_orchestrator_hygiene.py --include-workers --delete-transcripts`
6. 然后再开始第一条真实群测
7. 若群测结束后发现 hidden main / 已完成 worker 的 `main` session 被自动删除，不要手工补回；这是控制面为避免旧上下文跨 job 残留而主动轮转
- agent 绑定正确

如果这是 brownfield 迁移，而不是全新部署，迁移前备份范围至少包括：

- `~/.openclaw/openclaw.json`
- `~/.openclaw/v51-runtime-manifest.json`
- `~/.openclaw/teams/`
- `~/.config/systemd/user/v51-team-*`

brownfield 切换顺序固定为：

1. `internal_main`
2. `external_main`

一次只切一个 team，单组完成 clean redeploy、hygiene、单群验收后，再切下一组。

## 8. V5.1 首次上线必须做的会话卫生

只要是下面这些场景之一，就先做 hygiene 再测：

- 第一次上线
- 大改 `systemPrompt`
- 大改 hidden main / callback 协议
- 从旧版 team-orchestrator 配置迁移到 `V5.1 Hardening`
- 之前群里已经出现过协议残留、重复消息或顺序错乱

内部群示例：

```bash
python3 ~/.openclaw/tools/v5/v51_team_orchestrator_hygiene.py \
  --home ~/.openclaw \
  --group-peer-id oc_f785e73d3c00954d4ccd5d49b63ef919 \
  --team-key internal_main \
  --include-workers \
  --delete-transcripts
```

外部群示例：

```bash
python3 ~/.openclaw/tools/v5/v51_team_orchestrator_hygiene.py \
  --home ~/.openclaw \
  --group-peer-id oc_7121d87961740dbd72bd8e50e48ba5e3 \
  --team-key external_main \
  --include-workers \
  --delete-transcripts
```

作用：

- 清理 supervisor `group/main`
- 清理 worker `group/main`
- 砍断旧协议 transcript 污染

## 9. 第一次真实验收怎么测

强规则：

- 一次只测一个群
- 先内部群，后外部群
- 只 `@supervisor`
- 一次发一整条任务，不要拆开发

### 9.1 内部群测试词

```text
@奥特曼 请为我们做一个“3天限时促销冲刺”执行方案。

目标：
1. 3天成交 100 单
2. 总预算不超过 10000 元
3. 综合毛利率不低于 35%
4. 明天上午 10 点前可以直接开跑

请按团队模式完成，不要只给概念建议，必须最终给出：
- 运营执行节奏
- 每天的动作安排
- 预算分配与 ROI / 毛利红线
- 风险预案
- 最终统一收口
- 明日三件事
```

### 9.2 外部群测试词

```text
@奥特曼 请为一个景区客户设计“节前 3 天拉新促销方案”。

约束：
1. 目标新增付费订单 120 单
2. 总预算不超过 15000 元
3. 综合 ROI 不低于 2.2
4. 综合毛利率不低于 30%
5. 要兼顾投放、私域、客服和履约风险

请按团队模式完成，最后必须给出：
- 运营推进方案
- 财务测算与红线
- 最终可直接执行的统一方案
- 明日三件事
- 风险预案
```

### 9.3 群内正确顺序

你应该看到：

1. `【主管已接单｜jobRef】`
2. `【运营进度｜jobRef】`
3. `【运营结论｜jobRef】`
4. `【财务进度｜jobRef】`
5. `【财务结论｜jobRef】`
6. `【主管最终统一收口｜jobRef】`

### 9.4 不应该出现的内容

- 重复的 `【角色进度】` / `【角色结论】`
- 重复的 `【主管最终统一收口】`
- worker 自己直接发群消息导致的顺序混乱
- 长时间无回复
- 只跑一半不收口

### 9.5 双群并发最终验收 contract

当 `internal_main` 和 `external_main` 都完成单群验收后，再做一次双群并发 canary。最终通过标准固定为：

1. 新单编号全局唯一
2. 单群只出现一条 active job
3. 无重复阶段消息
4. 主管最终统一收口始终带 `jobRef`

## 11. 如果卡住了，优先怎么排查

### 11.1 先看运行健康

```bash
openclaw health
openclaw channels status --probe
```

### 11.2 再看 watchdog / reconcile

Linux / WSL2：

```bash
systemctl --user status v51-team-internal_main.service --no-pager -n 80
systemctl --user status v51-team-external_main.service --no-pager -n 80
systemctl --user list-timers --all 'v51-team-*' --no-pager
```

### 11.3 再手动补一次 resume-job

内部群：

```bash
python3 ~/.openclaw/tools/v5/v51_team_orchestrator_reconcile.py \
  --manifest ~/.openclaw/v51-runtime-manifest.json \
  --team-key internal_main \
  --openclaw-home ~/.openclaw \
  --openclaw-bin ~/.npm-global/bin/openclaw \
  resume-job \
  --stale-seconds 180
```

外部群：

```bash
python3 ~/.openclaw/tools/v5/v51_team_orchestrator_reconcile.py \
  --manifest ~/.openclaw/v51-runtime-manifest.json \
  --team-key external_main \
  --openclaw-home ~/.openclaw \
  --openclaw-bin ~/.npm-global/bin/openclaw \
  resume-job \
  --stale-seconds 180
```

### 11.4 这类现象说明什么

- supervisor 群第一条就裸 `NO_REPLY`
  - 先 hygiene，再 `resume-job`
- supervisor 群第一条没有进控制面，而是先 `read/exec/sessions_spawn`
  - 不要让用户连续重发；直接跑 `resume-job`。当前主线会从 supervisor group transcript 抢占最近未消费的真实用户消息补建单，并清理这条消息之后 supervisor 漂移出来的 subagent session
- worker 长时间不推进
  - 先检查 worker 是否真的提交了 `progressDraft / finalDraft / summary / details / risks / actionItems`
- worker `finalVisibleText` 提前写出其他角色章节、主管统一收口，或当前 `jobRef` 下出现 `sessions_spawn` 子代理
  - 视为协议违规；让 `resume-job` 拒绝该回调并清理 job scoped subagent/main session，再看重派后的新 worker 输出
- hidden main 里仍出现自由文本回调
  - 视为遗留协议漂移；当前主线不会再从 hidden main / worker transcript 文本恢复 callback，应先检查 worker 是否执行了 `callbackCommand(ingest-callback ...)`
- gateway 重启后持续刷 `delivery-recovery`
  - 优先清理 `~/.openclaw/delivery-queue/` 中遗留的旧坏消息，再重启 gateway
- DB `done` 但群里没最终收口
  - 看 `rollupVisibleSent` 是否为 `true`
- 手工 SSH 恢复时返回 `reconcile_already_running`
  - 说明当前 team 的 timer/另一份控制面实例已经持有独占锁；这是新的保护机制，不要并行再起第二份 `resume-job`

## 12. 建议你保留的回滚命令

任何改动前先备份：

```bash
ts=$(date +%Y%m%d-%H%M%S)
mkdir -p ~/.openclaw/backups/$ts
cp ~/.openclaw/openclaw.json ~/.openclaw/backups/$ts/openclaw.json
cp ~/.openclaw/v51-runtime-manifest.json ~/.openclaw/backups/$ts/v51-runtime-manifest.json 2>/dev/null || true
```

如果这次上线有问题，最小回滚：

```bash
cp ~/.openclaw/backups/<时间戳>/openclaw.json ~/.openclaw/openclaw.json
cp ~/.openclaw/backups/<时间戳>/v51-runtime-manifest.json ~/.openclaw/v51-runtime-manifest.json 2>/dev/null || true
openclaw config validate
openclaw gateway restart
```

如果这次同时替换了 `~/.openclaw/tools/v5` 下的脚本，也要一并回滚。
