# V5.1 Supervisor Rollup Template Design

## Goal

把 `【主管最终统一收口｜jobRef】` 从“字段拼接摘要”升级成“主管综合决策方案”。

这次调整的目标不是给商业场景单独做一套话术，而是沉淀一个可以复用于不同团队的通用主管收口骨架：

1. 能吸收多个 worker 的完整终案结论
2. 能体现主管的综合判断和取舍
3. 能落成一份可以直接执行的路线，而不是角色输出列表

## Problem

当前控制面在生成 rollup 时，主要消费的是 worker callback 里的：

- `summary`
- `details`
- `risks`
- `actionItems`

`finalVisibleText` 只作为兜底摘要来源，没有作为主管终案的核心输入。

这会带来三个问题：

1. 前面 worker 写得再完整，主管收口也会被压缩成 `- 运营：... - 财务：...` 这样的列表
2. 主管消息缺少“推荐采用哪条方案、为什么、怎么推进”的综合决策层
3. 模板被场景绑死后，很难复用到法务、交付、技术等其他群

## Design Principles

### 1. 通用骨架，不写死业务角色

主管收口固定采用以下结构：

1. `一、最终判断`
2. `二、建议采用的方案`
3. `三、整合依据`
4. `四、执行路线图`
5. `五、联合风险与红线`
6. `六、明日三件事`

其中“运营/财务/法务/技术”都只是当前 job 的输入角色，不是模板本身的固定字段。

### 2. 完整终案优先，结构字段辅助

每个 worker 的信息优先级改成：

1. `finalVisibleText`
2. `summary`
3. `details / risks / actionItems`

也就是说，主管综合方案的主要素材来自各角色完整终案正文；结构字段只负责补足摘要、风险和动作清单。

### 3. 主管负责综合，不负责复读

控制面输出应体现：

- 已吸收哪些角色判断
- 推荐的主方案是什么
- 如何把多角色结论串成一条可执行路线
- 哪些风险是统一红线

而不是把多个角色的字段逐条平铺。

## Rendering Strategy

### 一、最终判断

输出主管的总判断，说明已经综合哪些角色结论，并给出统一推进原则。

这部分使用固定句式，避免因为角色变化导致结构漂移。

### 二、建议采用的方案

从每个 worker 的 `finalVisibleText` 中提取前 1-2 条非标题正文，作为该角色被主管吸纳进主方案的核心判断。

呈现形式是“主管建议采用的联合方案”，而不是“角色原文搬运”。

### 三、整合依据

按角色输出：

- 已采纳该角色哪条核心结论
- 该角色对最终方案提供了什么约束/补充

这一节允许保留角色标签，因为这里本来就是解释综合依据。

### 四、执行路线图

优先使用 `actionItems`，不足时回退到 `details` 或 `finalVisibleText` 中的执行型句子。

输出仍然是统一路线图，但每条动作可以标注主要责任角色，方便复用到不同群。

### 五、联合风险与红线

继续聚合 `risks`，但统一表述成主管红线，不再只是“角色风险列表”。

### 六、明日三件事

从全体 `actionItems` 中提取前三条高优先动作。

如果不足三条，再从 `details` 中回填，确保该章节始终稳定存在且固定为三条。

## Reusability

这套设计对不同群通用，因为：

1. 模板不依赖固定角色名称
2. 角色只影响“整合依据”和“动作责任归属”的文案
3. 即便主管是法务、商业、交付或技术，都仍然需要“判断、方案、依据、路线、红线、明日动作”这套输出

## Acceptance Criteria

1. 主管最终收口不再只输出 `角色：summary/details` 列表
2. rollup 优先吸收 worker 的 `finalVisibleText` 正文
3. 输出结构固定为六段通用决策模板
4. “明日三件事”稳定存在且只有三条
5. 现有不同角色组合的 job 都能复用，不要求为每个群维护独立模板

## Finalized Outcome

在实际联调与群内验证后，这套模板又继续收敛为更强的“决策型主管稿”。

最终采用的章节为：

1. `一、最终结论`
2. `二、决策依据`
3. `三、最终方案`
4. `四、执行路线`
5. `五、风险红线`
6. `六、明日三件事`

这次终版收敛背后的经验是：

1. 主管稿不能只是“结构化”，还要像负责人在拍板
2. `details / risks / actionItems` 的数组必须在 callback sink 和 rollup parser 两层都正确归一化，否则最终群消息会出现 `['...']` 这样的脏格式
3. 好的终稿不是把 worker 长文再排一次版，而是把它们转译成：
   - 为什么这么定
   - 最终采用什么方案
   - 谁先做什么
   - 哪些红线不能碰

最终线上验证结果表明，这一版已经能在商业群与运维群里稳定输出“主管统一拍板稿”，比早期的摘要拼接版本明显更接近真实负责人发言。
